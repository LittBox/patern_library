# 前端样式偏好与踩坑总结

## 目标风格

### 整体气质
- 页面要有留白、呼吸感，不要元素挤在一起。
- 视觉上要轻一点、柔和一点，不要大黑块、重按钮、强压迫感。
- 信息层级要清楚，但不要靠又厚又重的边框来实现。
- 页面要更像产品界面，不像临时拼出来的后台页。

### 布局偏好
- 重要内容区域要居中，但不是所有元素都机械居中。
- 卡片、标题、分割线、按钮之间要共用同一条版心和对齐线。
- 移动端优先避免横向滚动，所有内容必须锁在视口内。
- 移动端能单列就单列，内容块尽量居中，避免左右失衡。
- 创作者中心卡片希望从左开始排，不要占满太宽。
- 创作者详情页头部希望“按钮在左，标题居中”，两者属于同一个头部容器。

### 导航偏好
- 移动端导航不要横向铺开。
- 移动端应使用左侧抽屉导航，由按钮呼出。
- 导航按钮和抽屉样式要轻，不要太厚重。

### 按钮与操作偏好
- 默认不喜欢大面积纯黑按钮，尤其在内容卡片里。
- 更偏好轻链接、幽灵按钮、图标操作，而不是厚重 CTA。
- 返回按钮只要 icon，不要“返回”文字。
- 返回 icon 要明显，但不需要边框盒子感太强。
- 创作者卡片底部的“详情”更适合做成右下角链接，而不是大按钮。
- 移动端最近作品操作区更适合 icon 或轻文字操作，不适合竖排大按钮。

### 图标偏好
- icon 要清晰、线条可见。
- 不喜欢黑底白字把 icon 吞掉的做法。
- 右箭头更适合轻量、细一点的图标，而不是 `->` 这种字符。

### 文案显示偏好
- 标题不要轻易换行，尤其像“创作者详情”这种短标题，应优先保持单行。
- 移动端多余说明文案应果断删除，保证信息聚焦。
- 操作文案如果导致拥挤，可以改成 icon 或 icon + 短词。

## JS 与交互约束

### 核心原则
- 调整前端时，优先改样式，不要轻易改动现有交互逻辑。
- 如果必须改 JS，也应尽量只做结构性最小改动，不影响原有业务流程。
- 任何视觉优化都不能以破坏上传、AI 生成、保存、删除、上架、价格修改为代价。

### 交互改动边界
- 不要因为样式调整随意改按钮的功能绑定。
- 不要因为更换 DOM 结构，导致原来的事件监听失效。
- 不要为了“看起来更简洁”就移除必要的可点击区域、`data-*` 标记或按钮 id。
- 不要把原本可点击的控件改成纯装饰元素。
- 如果把文字按钮改成 icon 或链接样式，必须保留原有行为、可点击性和可访问性。

### 请求与接口相关要求
- 前端样式改动不能影响接口请求路径和请求时机。
- 不要在改 UI 时顺手改动 API 调用基址，除非是在明确修复接口访问问题。
- 不要让移动端样式改动影响 `fetch` 成功与失败后的逻辑分支。
- 出现“占位图 / fallback 图”时，要优先判断接口失败原因，而不是只改表现层。

### 状态与数据要求
- 不要因为样式改动破坏本地状态恢复逻辑。
- 不要影响 `localStorage` / `IndexedDB` 的读写流程。
- 不要让“保存作品”“删除作品”“上架商品”“修改价格”这些操作因为换结构而失效。
- 如果改了 DOM 结构，要确认相关按钮仍然能正确找到对应作品 id、商品 id、pattern id。

### 移动端交互要求
- 移动端改版不能只顾视觉，必须同时检查点击热区是否够大、是否容易误触。
- 移动端导航、返回按钮、作品操作按钮都需要保留清晰可触达的点击区域。
- 不要为了极简把操作做得太隐蔽，用户仍然要一眼看出哪里能点。

### 可访问性要求
- icon 按钮不能只剩图形，必须保留 `aria-label` 或 `title`。
- 改成 icon 或链接样式后，仍要有清晰的 hover / focus / active 状态。
- 删除“返回”等文字后，要确保读屏和提示信息仍完整。

### 验证要求
- 每次改动交互相关 DOM 后，都应验证：
  - 导航还能切页
  - 返回按钮还能返回
  - 最近作品按钮还能打开详情或上架/改价
  - AI 页面生成按钮还能正常触发请求
  - 上传按钮还能正常选择文件并导入
- 如果只是改样式，但发现功能异常，优先检查是不是结构变更导致监听器失效。

## 已踩过的坑

### 1. 过度居中
- 一开始把“返回按钮 + 标题”都整体居中，导致按钮喧宾夺主。
- 后来又把按钮做成和标题绑定的整体，结果视觉上还是不自然。
- 真正合理的方案是：同一头部容器内，按钮固定左侧，标题独立居中。

### 2. 对齐线不统一
- 头部按钮、分割线、下方卡片没有共用同一条左边界，看起来很乱。
- 之前多次出现“按钮超出分割线”的问题，本质是按钮脱离了头部网格或头部宽度和内容宽度不一致。
- 结论：头部必须和内容卡片共用同一版心。

### 3. 被旧样式覆盖
- 某些新样式虽然写了，但仍然被旧规则覆盖。
- 典型问题：
  - 返回按钮被旧的 `position: absolute; left: 0;` 拉出容器。
  - 最近作品按钮仍继承 `.btn-tiny` 的黑底样式。
- 结论：以后改样式时要检查旧规则是否还在生效，而不是只看新增规则。

### 4. 移动端按钮过重
- 最近作品区域一度用了黑底按钮，视觉非常重，且在竖版下很丑。
- 纯文字竖排按钮也不好看。
- 结论：移动端局部操作优先用轻 icon 或轻链接，不要重复使用桌面端按钮样式。

### 5. AI 页面移动端溢出
- AI 生成功能页曾出现横向滑动、元素不居中、按钮撑出布局的问题。
- 原因包括：
  - 两列布局在移动端没有彻底收成单列。
  - 按钮有固定最小宽度。
  - 结果卡片和内容块没有限制在视口内。
- 结论：AI 页面必须单列、居中、禁横向滚动，并去掉会撑宽的最小宽度。

### 6. 卡片宽度过大
- 创作者中心卡片一度太宽，看起来像横幅，不像卡片。
- 用户更希望它像内容块，从左开始排，宽度更克制。
- 结论：卡片不应无节制拉宽，尤其单卡场景下要主动控制最大宽度。

### 7. 视觉太“硬”
- 黑底按钮、粗边框、过强阴影，经常会让页面显得笨重。
- 用户更偏好轻、透、柔和的视觉，而不是“重工业 UI”。
- 结论：后续样式调整优先使用：
  - 轻阴影
  - 弱边框
  - 透明/半透明底
  - 文字链接和 icon 操作

### 8. 样式改动连带影响交互
- 某些时候为了改按钮外观或布局，会改动 DOM 结构或 class，结果容易影响 JS 绑定。
- 这类问题不一定立刻表现为报错，但会表现成“按钮不响应”“切页失效”“详情打不开”。
- 结论：凡是改了交互元素的结构，都要同步检查事件绑定是否还在原节点上生效。


## 后续实现建议

- 新增 UI 时，先判断它属于“主 CTA”还是“次级操作”。
- 次级操作尽量不要直接复用 `.btn-primary`、`.btn-tiny` 这类强样式。
- 所有详情页头部都建议统一为：
  - 左侧返回 icon
  - 中间标题
  - 与正文共用版心
- 所有移动端页面都应检查：
  - 是否出现横向滚动
  - 标题是否异常换行
  - 卡片是否过宽
  - 操作按钮是否过重
  - icon 是否清晰可见
- 所有涉及交互元素的样式改动都应额外检查：
  - 事件绑定是否失效
  - `id` / `data-*` 是否还保留
  - 接口调用路径是否未被误改
  - fallback 和正常分支是否仍然可用
